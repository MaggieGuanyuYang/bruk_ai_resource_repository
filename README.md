# BR-UK AI Tools & Resources Repository

A GitHub Pages rebuild of the [Behavioural Research UK AI Tools and Resources Repository](https://usher.ed.ac.uk/behavioural-research-uk/learning-resources/ai-tools-and-resources-repository), rehoused as a standalone, behavioural-researcher-friendly site.

**Live site:** _deploy this repo to GitHub Pages; see below._

## How the site is built

Content lives in YAML/markdown under `content/`. A small Python build script
(`build.py`) renders the content into the published `*.html` files at the
project root using the templates in `templates/`. GitHub Pages serves those
HTML files directly.

```
.
├── content/                        ← edit this to change content
│   ├── _site.yml                   Brand, nav, footer, fonts
│   ├── home.yml                    Home (index.html)
│   ├── about.yml                   BR-UK Statement
│   ├── section-1/
│   │   ├── _index.yml              Living Guide overview
│   │   ├── literature-review.yml
│   │   ├── hypothesis-generation.yml
│   │   ├── data-collection.yml
│   │   ├── data-analysis.yml
│   │   └── writing-reporting.yml
│   ├── section-2/
│   │   ├── _index.yml              Ethics overview
│   │   ├── disclosure.yml
│   │   ├── publishers.yml          Publisher policies
│   │   ├── uk-funders.yml          Funder policies
│   │   ├── biases.yml
│   │   ├── privacy.yml
│   │   ├── sustainability.yml
│   │   └── key-regulations.yml
│   ├── section-3.yml               General AI learning resources
│   ├── section-4.yml               BR-UK AI Advice Sessions
│   └── section-5.yml               Disclaimer & contact
├── templates/                      ← edit this to change layout
│   ├── _base.html                  Head, header, footer chrome
│   ├── home.html                   Home page (hero + section grid)
│   └── page.html                   Every other page
├── build.py                        ← run this to regenerate HTML
├── build-requirements.txt          PyYAML, Jinja2, Markdown
├── SITE-STRUCTURE.md               Per-page section map (read first)
├── REVISION-TEMPLATE.md            Fill this in for monthly updates
├── index.html                      ← GENERATED — do not edit
├── about.html                      ← GENERATED
├── section-1/*.html                ← GENERATED
├── section-2/*.html                ← GENERATED
├── section-3/4/5.html              ← GENERATED
├── assets/
│   ├── css/style.css               Site stylesheet (edit this)
│   └── js/main.js                  Mobile-menu JS (edit this)
├── .nojekyll                       Tells GitHub Pages to skip Jekyll
└── README.md
```

The published `*.html` files **are committed to the repo** (so GitHub Pages
serves them with no build step), but they are **regenerated by `build.py`** —
do not hand-edit them. See [SITE-STRUCTURE.md](SITE-STRUCTURE.md) for the
per-page YAML key map and [REVISION-TEMPLATE.md](REVISION-TEMPLATE.md) for
the monthly-update workflow.

## Editing content

```bash
pip install -r build-requirements.txt   # one-time setup
# 1. Edit YAML under content/
# 2. Regenerate HTML
python3 build.py
# 3. Preview locally
python3 -m http.server 8080
# 4. Verify in browser, then commit BOTH the YAML and the regenerated *.html
```

Common edits and where to make them:

- **Add a new tool to a stage** → append a `{ name, url, desc, … }` entry to the
  relevant `tool_grid:` list in the page's YAML (e.g.
  `content/section-1/literature-review.yml`).
- **Fix a broken external link** → grep `content/` for the URL and replace.
- **Change a heading** → edit the matching `h2:` / `h3:` block. If the heading
  has an anchor, also update any `quicknav.items[]` that link to it.
- **Update the teal colour** → edit `--teal-*` custom properties at the top of
  `assets/css/style.css`. (No build step needed for stylesheet edits.)
- **Add a new sub-page** → create a new YAML under `content/<section>/`, add
  a `subpage_list` entry on the section's `_index.yml`, and update the prev/next
  pagers on adjacent sub-pages. Page IDs are derived from the path
  (`content/section-1/foo.yml` → `section-1.foo`); `ref:` everywhere uses the
  page ID, never a relative URL.

### Build-script flags

```bash
python3 build.py                  # rebuild every page
python3 build.py home             # rebuild a single page (by page ID)
python3 build.py --check          # dry-run; exits non-zero if any output
                                  #   would change. Use as a CI / pre-commit gate.
python3 build.py -q               # suppress per-page log lines
```

## Deploying to GitHub Pages

1. Create a new GitHub repository (e.g. `bruk-ai-repository`).
2. Push this directory to the repo:
   ```bash
   git init
   git add .
   git commit -m "Initial import of BR-UK AI Tools & Resources Repository"
   git branch -M main
   git remote add origin https://github.com/<your-user>/<repo-name>.git
   git push -u origin main
   ```
3. On GitHub, go to **Settings → Pages**.
4. Under **Build and deployment → Source**, choose **Deploy from a branch**, pick `main` and `/ (root)`, then save.
5. Wait a minute, then visit `https://<your-user>.github.io/<repo-name>/`.

The `.nojekyll` file prevents GitHub from processing the site through Jekyll — which is what you want here, since the site is plain HTML/CSS/JS and contains no Liquid templates.

## Accessibility

- Semantic HTML (landmarks, headings, lists, breadcrumbs).
- Skip-to-content link on every page.
- Visible focus outlines, 4.5:1+ contrast for body text against backgrounds.
- Keyboard-navigable mobile menu.
- Respects `prefers-reduced-motion`.

## Browser support

Targeted at evergreen browsers: Chrome, Firefox, Safari, Edge (current releases). No polyfills or transpilation required.
